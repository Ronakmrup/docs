## Introduction

Sanas Noise Cancellation (NC) provides AI-powered noise cancellation that extracts voices from noisy signals in real-time across any acoustic conditions. With low latency and minimal computing cost, NC delivers clean speech waveforms suitable for human communication.

Designed for real-time inference, NC works in both streaming (live audio) and offline (recorded files) scenarios, making it versatile for any voice communication application.

## Model Selection Guide

<Steps>
  <Step title="Identify your scenario">
    **How many speakers do you need to preserve?**

    - Multiple speakers → Use **ST** (Standard)
    - Single speaker → Continue to Step 2
  </Step>
  <Step title="What's your microphone type?">
    **For single speaker scenarios:**

    - Headset/close-proximity mic (~30cm) → Use **VI-N** (Nearfield)
    - Phone/VoIP (8kHz) → Use **VI-GT** (Telephony)
    - General microphone (~1m) → Use **VI-G** (General)
  </Step>
  <Step title="Configure in SDK">
    Set the model name in your audio parameters:

    ```python
    audio_params.modelName = "VI-G"  # Your chosen model
    audio_params.sampleRate = 16000   # Appropriate sample rate
    ```
  </Step>
</Steps>

## Latency & Performance

| Metric            | Value      | Description                         |
| :---------------- | :--------- | :---------------------------------- |
| Streaming Latency | ~40ms      | Per-chunk processing time           |
| Throughput        | Real-time+ | Processes faster than real-time     |
| Concurrency       | Unlimited  | Limited only by your infrastructure |

### Real-Time Call Latency Budget

```text
Speaker → Mic capture (5-10ms)
        → Network (20-50ms)
        → Sanas NC (~40ms)
        → Encoder (10-20ms)
        → Network (20-50ms)
        → Decoder (10-20ms)
        → Speaker output (5-10ms)
Total: 110-200ms end-to-end
Sanas contribution: ~40ms (20-36% of total latency)
```

## Integration Examples

### WebRTC Integration

```python
# In your WebRTC audio processing pipeline
def process_audio_frame(frame):
    # Get audio samples from WebRTC frame
    audio_samples = frame.to_ndarray()
    
    # Process with Sanas NC
    clean_samples = processor.ProcessSamples(audio_samples.tolist())
    
    # Create new frame with clean audio
    clean_frame = av.AudioFrame.from_ndarray(
        np.array(clean_samples), 
        format='s16', 
        layout='mono'
    )
    return clean_frame
```

### SIP/VoIP Integration

```python
# For telephony applications with 8kHz audio
audio_params.modelName = "VI-GT"

    audio_payload = rtp_packet.payload
    clean_audio = processor.ProcessSamples(audio_payload)
    # Forward clean audio to recording/monitoring
```

## Best Practices

<CardGroup cols={2}>
  <Card title="Model Selection" icon="sliders">
    **Do:**

    - Match model to acoustic environment
    - Consider microphone distance

    - Use VI-G for conference rooms (use ST)

## Troubleshooting

<AccordionGroup>
  <Accordion title="Voice sounds robotic or distorted" icon="triangle-exclamation">
    **Possible causes:**

    - Wrong model variant for scenario
    - Sample rate mismatch
    - Multiple NC stages in pipeline

    **Solutions:**

    - Verify model matches use case (e.g., ST for multi-speaker, not VI-G)
    - Check sample rate matches input audio
    - Remove any other noise cancellation before Sanas
  </Accordion>
  <Accordion title="Background noise not fully removed" icon="triangle-exclamation">
    **Possible causes:**

    - Model variant doesn't match scenario
    - Microphone too far from speaker

    **Solutions:**

    - For single speaker, use VI-G (not ST)
    - Check microphone distance matches model expectations:
      - VI-N: ~30cm
      - VI-G: ~1m
      - ST: ~3m (preserves all speakers)
  </Accordion>
  <Accordion title="Background voices removed when they shouldn't be" icon="triangle-exclamation">
    **Cause:** Using Voice Isolation model instead of Standard

    **Solution:** Use ST model for multi-speaker scenarios

    ```python
    # For conference room with multiple speakers
    audio_params.modelName = "ST"  # Not VI-G, VI-N, or VI-GT
    ```
  </Accordion>
  <Accordion title="Processor creation fails" icon="triangle-exclamation">
    **Common issues:**

    - Invalid model name (check spelling: `ST`, `VI-G`, `VI-GT`, `VI-N`)
    - Invalid sample rate for model
    - Network connectivity

    **Verify configuration:**

    ```python
    print(f"Model: {audio_params.modelName}")
    print(f"Sample rate: {audio_params.sampleRate}")
    
    # Check model-specific limits:
    # ST, VI-G, VI-N: 8, 16, or 24 kHz
    # VI-GT: 8 kHz only
    ```
  </Accordion>
</AccordionGroup>
